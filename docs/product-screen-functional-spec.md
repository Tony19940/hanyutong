# Bunson老师界面功能说明与需求澄清

## 1. 文档目的

本文档基于当前 `Bunson老师` 项目已实现版本整理，目标是把产品的每个界面、每个核心功能、当前行为和后续需求澄清统一写清楚，便于后续：

- 继续做产品设计
- 拆分研发任务
- 对齐 UI / 内容 / 技术实现
- 避免需求只停留在口头沟通

本文档以“当前产品事实”为基础，不是理想态蓝图。

## 2. 产品定位

“Bunson老师”是一款面向柬埔寨中文初学者的 Telegram Mini App / 移动端中文学习产品。核心价值不是“讲很多知识”，而是：

- 帮用户快速理解词和句子
- 帮用户在生活场景里敢开口说中文
- 用高棉语降低理解门槛
- 用更像真人老师的 AI 对话体验提高留存

当前产品的四条主路径为：

- 学习：滑卡背词
- 测验：轻游戏化复习
- 对话：语音消息式口语训练
- 我的：学习成果、收藏、分享

## 3. 全局产品规则

### 3.1 导航结构

- 底部 Tab 共 4 个：
  - 学习
  - 测验
  - 对话
  - 我的
- 收藏不再独立占一个 Tab，而是收纳进“我的”

### 3.2 用户身份与资料

- 优先读取 Telegram 用户资料
- 头像优先使用 Telegram 头像
- 若 Telegram 头像不可用，回退到本地默认头像
- 昵称优先展示 Telegram username / display name

### 3.3 音频原则

- 固定词库音频采用预生成音频
- 对话页采用 `ASR + Doubao-Seed-1.6-flash + TTS`
- 固定教学消息优先显示：
  - 中文
  - 高棉语翻译
- 动态 AI 回复当前以中文为主，不强制实时高棉语翻译

### 3.4 设备与布局原则

- 主场景是手机
- 桌面端预览也应模拟手机窗口感
- 核心操作必须在手机单手操作下成立

## 4. 界面一：登录页

### 4.1 页面目标

让用户完成激活码登录，并进入学习主流程。

### 4.2 入口

- 首次进入应用
- 登录态失效
- 退出后重新进入

### 4.3 主要模块

#### 模块 A：激活码输入框

功能描述：

- 用户输入激活码
- 输入过程中自动格式化为 `HYT-XXXX-XXXX-XXXX`
- 支持键盘回车提交

当前行为：

- 输入格式会自动转大写并插入分隔符
- 无效输入会提示错误

需求澄清：

- 激活码是当前唯一登录方式，不做手机号 / 邮箱登录
- 后续若接 Telegram 一键授权，应该是“补充方式”，不是替代激活码

#### 模块 B：开始学习按钮

功能描述：

- 提交激活码
- 创建用户会话
- 登录成功后进入主应用

当前行为：

- 按钮承担主 CTA
- 登录成功后直接进入学习页

需求澄清：

- 当前设计强调“激活后进入学习”，而不是注册流程
- 文案应保持简单直接，不增加多余说明

#### 模块 C：购买/联系支持入口

功能描述：

- 为没有激活码的用户提供购买或联系渠道

当前行为：

- 提供 Telegram 联系入口

需求澄清：

- 该入口当前承担“售前支持”角色
- 后续如果加入价格页，可把该入口升级为购买路径

## 5. 界面二：学习页（首页）

### 5.1 页面目标

让用户通过最低认知负担的方式快速接触中文词汇，完成“看词、听词、看例句、标记状态”的循环。

### 5.2 入口

- 底部 Tab：学习
- 登录后的默认主页面

### 5.3 主要模块

#### 模块 A：学习进度摘要

功能描述：

- 展示已学进度
- 让用户知道当前整体完成度

当前行为：

- 展示已学习数量 / 总量 / 百分比

需求澄清：

- 这里的目标是轻量提醒，不是重度数据分析面板
- 不应放太多解释性文案

#### 模块 B：单词卡片

功能描述：

- 每次展示一个单词学习单元
- 包含：
  - 中文
  - 拼音
  - 高棉语释义
  - 例句 1
  - 例句 2（如果有）
- 用户可以听词和例句发音

当前行为：

- 整张卡都支持左右滑动
- 自动朗读：
  - 单词
  - 例句
- 例句可单独点击播放

需求澄清：

- 卡片是“最小学习单元”
- 当前不追求一张卡承载很多知识点
- 优先保证“看得懂、听得到、滑得动”

#### 模块 C：左滑 / 右滑操作

功能描述：

- 左滑：学会了
- 右滑：收藏

当前行为：

- 操作后立即切换到下一张卡

需求澄清：

- 左右滑是首页核心交互，应保留
- 不建议在首页改成闯关式主玩法
- 测验和对话承担更深层练习，不与滑卡主功能冲突

#### 模块 D：音频播放

功能描述：

- 用户可听标准发音
- 自动播放帮助减少点击成本

当前行为：

- 预生成音频优先
- 单词和例句都可朗读

需求澄清：

- 首页音频目标是“快速输入”
- 不在首页做发音评分

## 6. 界面三：测验页

### 6.1 页面目标

用更轻量、更像小游戏的方式让用户复习已学词句，避免只停留在被动浏览。

### 6.2 入口

- 底部 Tab：测验

### 6.3 主要模块

#### 模块 A：关卡式题目流程

功能描述：

- 每轮为一个小测验
- 由多个题型组成

当前行为：

- 一轮通常围绕 5 个词展开
- 题型包括：
  - 看高棉语选中文
  - 看中文选意思
  - 看拼音选词
  - 听音选义
  - 例句相关题型

需求澄清：

- 测验页定位是“复习强化”
- 不是完整课程页
- 不应加入过多复杂讲解

#### 模块 B：分数、连击、生命值

功能描述：

- 增强反馈感与游戏感

当前行为：

- 有 hearts / score / combo
- 答题正误会触发音效和震动

需求澄清：

- 游戏化以“即时反馈”为主
- 不做复杂养成体系第一版

#### 模块 C：答题结果反馈栏

功能描述：

- 答题后显示正确 / 错误反馈

当前行为：

- 已调整为底部吸附，停靠在 Tab 栏上方

需求澄清：

- 反馈不应遮挡题目主内容
- 文案应短促、结果导向

#### 模块 D：例句发音回放

功能描述：

- 答题后用户可听相关例句

当前行为：

- 正确/错误后会触发对应例句播放

需求澄清：

- 该模块帮助用户把“题目”与“真实用法”重新绑定
- 不要求每道题都做长反馈解释

## 7. 界面四：对话页

### 7.1 页面目标

让用户像和真人老师发语音消息一样练中文口语，而不是像使用一个机械系统。

### 7.2 入口

- 底部 Tab：对话

### 7.3 当前产品定位

当前对话页不是开放闲聊，而是“受控场景练习”：

- 每天推荐 3 个话题
- 用户点击一个话题进入当天练习
- AI 老师以语音消息形式带练

### 7.4 主要模块

#### 模块 A：老师会话头部

功能描述：

- 像聊天软件一样展示：
  - 老师头像
  - 老师姓名
  - 在线状态

当前行为：

- 头像使用 `Bunson老师` 的指定图片
- 状态平时显示“在线”
- AI 回复等待时显示“正在输入…”

需求澄清：

- 目标是建立“在和一个老师聊天”的感受
- 不应出现强系统感文案

#### 模块 B：每日话题卡片

功能描述：

- 每天从 50 个话题中推荐 3 个
- 以大圆角卡片置中展示
- 用户点击任一话题开始当天练习

当前行为：

- 卡片显示标题与副标题
- 点击后进入准备态并开始生成起始消息

需求澄清：

- 录音按钮不承担“开始话题”功能
- 话题必须显眼，且点击后有明确反馈动画
- 后续可加入推荐去重逻辑和“继续上次”逻辑

#### 模块 C：消息列表

功能描述：

- 像 Telegram / 微信一样展示消息流
- 支持：
  - 老师语音消息
  - 老师文字
  - 老师高棉语翻译
  - 用户语音消息
  - 时间戳

当前行为：

- 最新消息应始终完整出现在底部录音栏上方
- 消息列表可手动上下滚动查看历史
- 新消息出现时有轻微过渡动画

需求澄清：

- 历史消息必须可回看
- 最新消息优先级高于顶部空间完整性
- 不应出现“最新消息被底部录音栏遮挡”的情况

#### 模块 D：语音消息播放状态

功能描述：

- 点击任意语音消息进行播放
- 用户需要知道“当前正在播放哪一条”

当前行为：

- 当前播放消息显示波形进度和播放态

需求澄清：

- 语音消息必须像聊天软件里的语音条
- 不应只靠耳朵判断当前播放项

#### 模块 E：录音控件

功能描述：

- 底部固定录音主按钮
- 点击开始录音
- 再点击结束录音并发送

当前行为：

- 计时器已融入按钮本体
- 录音中按钮显示红色与波形动画

需求澄清：

- 录音按钮只有两个状态：
  - 开始录音
  - 结束录音
- 不再承担“开始话题”职责

#### 模块 F：语音识别与等待态

功能描述：

- 用户发送语音后，系统需要一段处理时间

当前行为：

- 不再在用户消息上显示“正在识别”
- 改为老师侧显示“正在输入…”
- 同时出现等待中的老师气泡

需求澄清：

- 用户视角应感觉“老师正在回复”
- 不应看到太多后台术语

#### 模块 G：教学引导节奏

功能描述：

- 每个话题开始时由老师引导进入练习

当前行为：

- 已从原来的 3 条起始消息压缩为更短节奏
- 当前更接近：
  - 话题标签
  - 一条简短开场
  - 一条目标句

需求澄清：

- 对话页的核心是跟读，不是讲解
- 开场不应堆太多消息
- 更像真人老师自然带一句，而不是系统播课

#### 模块 H：高棉语辅助

功能描述：

- 帮初学者理解固定教学内容

当前行为：

- 固定教学消息显示：
  - 中文
  - 高棉语

需求澄清：

- 当前优先覆盖固定脚本
- 动态 AI 回复是否实时翻译为高棉语，后续再评估
- 基于成本与稳定性，更适合“预生成固定翻译 + 动态内容视情况缓存”

### 7.5 话题系统要求

当前已扩充到 50 个话题，原则上应满足：

- 覆盖高频生活场景
- 适合中文初学者
- 每个话题是完整可练的微场景

需求澄清：

- 当前“每日话题”应是真正轮换推荐，而不是固定 3 个永久不变
- 后续应增加：
  - 推荐去重
  - 新旧话题轮换
  - 继续上次练习
  - 难度分层

## 8. 界面五：我的页面

### 8.1 页面目标

承载用户身份、学习成果和次级操作入口。

### 8.2 入口

- 底部 Tab：我的

### 8.3 主要模块

#### 模块 A：头像和昵称

功能描述：

- 展示当前用户身份

当前行为：

- 优先 Telegram 头像
- 失败则回退默认头像
- 展示 display name / username

需求澄清：

- 头像模块的重点是“可信身份感”
- 不做复杂编辑资料功能第一版

#### 模块 B：学习统计

功能描述：

- 展示：
  - 已学词汇
  - 学习时长
  - 掌握度

当前行为：

- 使用统计卡片呈现

需求澄清：

- 统计应帮助建立成就感
- 不应变成复杂数据看板

#### 模块 C：连续学习与成就感

功能描述：

- 展示 streak 和近期学习活跃情况

当前行为：

- 有连续学习卡片和近 7 天点阵

需求澄清：

- 这是轻量激励，不是养成系统

#### 模块 D：分享成果

功能描述：

- 打开分享小票
- 用于保存和传播学习成果

当前行为：

- 弹出分享层
- 支持保存图片 / Facebook 分享

需求澄清：

- 分享是传播工具，也是成就强化工具
- 小票设计应强调可读性与完成感

#### 模块 E：收藏词库入口

功能描述：

- 进入收藏词列表

当前行为：

- 收藏词库收纳在“我的”中，而不是独立 Tab

需求澄清：

- 收藏的定位是“稍后复习”
- 不是主路径首页

#### 模块 F：退出登录

功能描述：

- 清除登录态

当前行为：

- 作为页面底部主动作之一

需求澄清：

- 当前产品设置较少，退出登录是少数强动作入口之一

### 8.4 布局要求

需求澄清：

- 我的页的一屏点击完整性很重要
- 分享 / 收藏 / 登录状态等核心入口应尽量一屏完成
- 页面不应被过多冗余设置项稀释

## 9. 界面六：收藏页

### 9.1 页面目标

让用户查看和复习被右滑收藏的词。

### 9.2 入口

- 我的页 -> 收藏词库

### 9.3 主要模块

#### 模块 A：收藏列表

功能描述：

- 展示所有已收藏词条

当前行为：

- 列表项展示：
  - 中文
  - 拼音
  - 高棉语
  - 播放按钮

需求澄清：

- 列表承担“快速浏览和定位”作用
- 不应承担复杂讲解功能

#### 模块 B：词卡详情

功能描述：

- 点开某个收藏词后进入词卡详情

当前行为：

- 复用 `WordCard`
- 支持播放
- 支持左滑移出收藏 / 标记学会

需求澄清：

- 收藏页的详情体验应和首页词卡保持一致
- 避免两个地方同一词显示逻辑不同

#### 模块 C：空状态

功能描述：

- 当没有收藏词时，提示用户如何使用收藏

当前行为：

- 有空状态说明

需求澄清：

- 空状态应简短，不要过度说明

## 10. 界面七：分享弹层

### 10.1 页面目标

让用户把学习成果以视觉化方式保存或分享。

### 10.2 入口

- 我的页 -> 分享成果

### 10.3 主要模块

#### 模块 A：学习小票展示区

功能描述：

- 展示“学习小票”视觉
- 包含学习成果信息

当前行为：

- 使用 receipt 风格组件
- 强调可保存和可分享

需求澄清：

- 重点是“可读性”和“仪式感”
- 不应为了炫技而牺牲文字可辨识度

#### 模块 B：保存图片

功能描述：

- 把当前成果图保存为图片
